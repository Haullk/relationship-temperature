# 双边关系趋势产品需求文档

## 1. 产品概述

### 1.0 产品工作名

产品工作名：关系温度计。

该名称用于首版需求、设计和工程沟通，后续可在品牌阶段调整。

### 1.1 产品一句话

点选候选池内任意两方合法组合，看到近 90 天双边关系冷暖趋势；点击趋势线上的重要趋势段，知道这段关系为什么变。

### 1.2 产品定位

本产品是面向大众读者的“双边关系解释器”。它参考 PizzINT 这类简洁监控式界面，但不把首版定位为威胁预警或专业情报系统，而是帮助用户快速理解：

- 两方关系最近是在改善、恶化还是保持平稳。
- 关系变化发生在哪些时间点。
- 哪些事件类型和重点报道线索可能推动了这些变化。

PizzINT 是一个面向地缘关系和风险监控的简洁仪表盘类产品。本产品只参考其“卡片 + 趋势图 + 方法说明”的清晰界面组织方式，不复制其威胁指数定位。

### 1.3 核心体验

用户进入页面后，先看到重点关系卡片；选择任意一组候选关系后，页面展示单一国家对的关系温度趋势图。趋势线上自动标出 最多 6 个重要趋势段。用户点击任意趋势段后，看到简明解释、主要驱动事件类型和重点报道线索。

### 1.4 目标用户

首版优先服务大众读者，包括对国际新闻、地缘政治、宏观趋势感兴趣但不了解 GDELT、CAMEO、Goldstein 等专业概念的用户。

产品文案应避免默认暴露技术术语，优先使用“关系温度”“偏合作”“偏冲突”“改善”“恶化”“主要驱动事件”等易懂表达。技术口径放入折叠式说明。

## 2. 首版范围

### 2.1 首版必须支持的关系

首页重点展示以下关系：

| 关系 | 说明 |
| --- | --- |
| 中美 | 中国 - 美国 |
| 中俄 | 中国 - 俄罗斯 |
| 中欧 | 中国 - 欧盟/欧洲 |
| 中日 | 中国 - 日本 |
| 中印 | 中国 - 印度 |
| 美伊 | 美国 - 伊朗 |
| 美俄 | 美国 - 俄罗斯 |

### 2.2 候选对象池

首版下拉选择器中的候选对象不超过 10 个：

| 界面名称 | 内部代码/口径 |
| --- | --- |
| 中国 | `CHN` |
| 美国 | `USA` |
| 俄罗斯 | `RUS` |
| 欧盟/欧洲 | 聚合对象，见 2.3 |
| 日本 | `JPN` |
| 印度 | `IND` |
| 伊朗 | `IRN` |
| 中国台湾 | `TWN` |
| 英国 | `GBR` |
| 德国 | `DEU` |

首版中的“任意两国”应解释为“候选池内任意两方合法组合”，不是全球所有国家任意选择。

### 2.3 中欧口径

“中欧”在首版中不是单一国家对，而是聚合对象：

```text
EUR + GBR + DEU + FRA + ITA + ESP + NLD
```

界面统一显示为“欧盟/欧洲”。其中：

- `EUR` 用于覆盖 GDELT 中被编码为欧盟或欧洲机构的事件。
- `GBR/DEU/FRA/ITA/ESP/NLD` 用于覆盖中国与主要欧洲国家之间的事件。
- 若用户选择“欧盟/欧洲”与其他对象组合，后台按上述代码集合进行聚合计算。
- 为避免聚合对象与成员对象重复计数，首版合法组合中排除 `europe_gbr` 和 `europe_deu`。

### 2.4 首版不做

首版不包含以下能力：

- 不做账号系统、登录、订阅、提醒、告警。
- 不做 CSV、图片或报告导出。
- 不抓取外部网页标题、正文或摘要。
- 不接入 AI 自动总结。
- 不要求历史数据回填到一年或多年。

这些能力可作为后续路线规划。

## 3. 页面与交互需求

### 3.1 页面结构

首版采用“卡片 + 图 + 解释”的单页结构：

1. 顶部：产品标题、简短说明、数据更新时间。
2. 重点关系卡片区：展示中美、中俄、中欧、中日、中印、美伊、美俄。
3. 国家对选择器：两个下拉框，从候选对象池中选择两方。
4. 单对关系趋势图：展示所选关系近 90 天关系温度。
5. 趋势段解释区：点击趋势线标记后展示原因解释。
6. 折叠式方法说明：解释数据来源、分数含义、限制与误差。

### 3.2 重点关系卡片

每张卡片参考 PizzINT 的简洁监控式表达，但文案聚焦“关系冷暖”。

卡片至少展示：

- 关系名称，例如“中美”。
- 当前关系温度，0-100。
- 当前状态：偏合作、接近中性、偏冲突。
- 近 7 日变化：改善、恶化、平稳。
- 小型趋势线或趋势缩略图。
- 与当前关系温度一致的分区配色。

卡片当前状态采用 3 档文字，颜色仍按 5 档温度分区渲染：

- `明显偏合作`、`偏合作` 折叠显示为“偏合作”。
- `接近中性` 显示为“接近中性”。
- `偏冲突`、`明显偏冲突` 折叠显示为“偏冲突”。
- 卡片背景、状态标签和迷你趋势线末端仍使用 5 档分区色。

点击卡片后，主趋势图切换到对应关系。

### 3.3 国家对选择器

选择器由两个下拉框组成：

- 左侧选择对象 A。
- 右侧选择对象 B。
- 默认展示首页重点关系中的第一组，建议为“中美”。
- 不允许 A 与 B 相同。
- 选择“欧盟/欧洲”时，后台使用聚合口径计算。
- 不允许选择“欧盟/欧洲”与其首版候选池成员“英国”“德国”组成关系。
- 用户切换国家对后，页面 URL 应同步更新，便于复制和分享。

### 3.3.1 URL 分享

首版支持通过 URL 参数直接打开某一组关系。

URL 规则：

```text
/trend?pair=chn_usa
```

要求：

- `pair` 使用小写对象代码，以 `_` 连接。
- `pair` 校验时先按两个对象代码的字母序生成规范组合 ID，因此 `/trend?pair=usa_chn` 与 `/trend?pair=chn_usa` 等价；前端展示和分享时统一回写规范组合 ID。
- 打开 `/trend?pair=chn_usa` 时，页面自动选中“中美”关系。
- 用户点击重点关系卡片或修改下拉框后，URL 同步更新，但不刷新页面。
- 分享按钮复制当前 URL。
- 如果 `pair` 参数缺失，默认打开 `chn_usa`。
- 如果 `pair` 格式无效、对象不在候选池中，或组合不在合法组合列表中，回退到默认关系，并展示轻量提示：“链接中的关系组合暂不可用，已切换到默认关系。”
- 如果 `pair` 合法但缓存暂无该组合数据，展示该组合的无数据空态，不回退默认关系。
- “欧盟/欧洲”的 URL 对象代码使用 `europe`，例如 `chn_europe`。
- `europe_gbr` 和 `europe_deu` 不属于合法 URL 组合。
- 规范组合 ID 不决定界面显示顺序，例如 `rus_usa` 仍可显示为“美俄”。

### 3.4 关系趋势图

主趋势图只展示当前选中的一组关系，不默认叠加多条线。

图表要求：

- 横轴：日期，覆盖当前数据库可用的近 90 天。
- 纵轴：关系温度，范围 0-100。
- 50 为中性线。
- 提供 90 日、30 日、15 日图表范围切换；范围切换只影响图表显示和可选趋势段，不改变“当前温度”的最新日期口径。
- 鼠标 hover 趋势线时展示对应日期和温度，并显示垂直到 X 轴日期位置的辅助线。
- 0、25、50、75、100 参考线统一使用灰色虚线，避免参考线颜色和趋势语义混淆。
- 高于 50 表示更偏合作。
- 低于 50 表示更偏冲突。
- 趋势线使用 14 日平滑。
- 自动标出 最多 6 个重要趋势段。

趋势段标记应清晰可点击。点击后，解释区更新为对应趋势段的解释。

### 3.4.1 关系温度分区配色

关系温度必须通过颜色和文字共同表达，让用户看一眼卡片和图表就能判断关系冷暖。

首版采用五档分区：

| 温度范围 | 状态文案 | 颜色语义 |
| --- | --- | --- |
| 0-35 | 明显偏冲突 | 冲突红 |
| 35-45 | 偏冲突 | 警示橙 |
| 45-55 | 接近中性 | 中性灰 |
| 55-65 | 偏合作 | 合作蓝 |
| 65-100 | 明显偏合作 | 合作绿 |

展示要求：

- 重点关系卡片背景、状态标签和迷你趋势线末端可使用对应分区色。
- 重点关系卡片的文字状态按 3 档折叠展示，但颜色仍按本表 5 档分区渲染。
- 主趋势图中的当前值、趋势段和图例应保持同一分区色语义。
- 颜色只作为辅助表达，必须同时展示状态文案，避免用户仅靠颜色理解。
- 50 中性线保持固定，帮助用户判断偏合作或偏冲突。

### 3.5 趋势段解释区

用户点击趋势段后，展示简明解释，而不是研究表格。

解释区包含：

1. 一句话模板解释。
2. 分数变化。
3. 变化期与基准期对比窗口。
4. 前三类驱动事件。
5. 3-6 条重点报道线索。

示例文案：

```text
截至该趋势段，中美关系温度下降。最近 7 天相比前 7 天，“关系降级”“反对/批评”“威胁”等偏冲突事件占比上升，相关报道提及也增加。
```

解释必须使用谨慎措辞，例如“可能推动”“媒体事件信号显示”“相关报道线索”，不能写成确定因果。

### 3.6 重点报道线索

首版只使用本地数据库已有字段，不抓取新闻页面。

重点报道卡片展示：

- 日期。
- 来源域名。
- URL 线索标题，即从 URL slug 中提取的可读文本。
- 事件类型，例如“外交合作”“关系降级”“威胁”。
- 影响方向，例如“偏合作”“偏冲突”。
- 原始链接。

如果 URL 线索标题不可读，则展示来源域名和链接，不生成虚假标题。

### 3.7 方法说明

主界面默认隐藏 GDELT、Goldstein、CAMEO 等术语。页面提供折叠式“数据来源与限制”说明。

说明内容必须包含：

- 数据来自本地 MapNews 共享数据库中的 GDELT 结构化事件。
- 关系温度是媒体事件信号，不是官方外交结论。
- 当前库没有新闻正文和标题，重点报道仅展示来源链接线索。
- 自动事件编码可能存在地名、组织、人物识别误差。
- 趋势解释是归因线索，不是严格因果证明。

### 3.8 状态设计

页面必须覆盖加载态、空态和错误态，不能只设计理想数据状态。

#### 3.8.1 加载态

- 页面进入后立即显示重点关系卡片、图表和解释区的骨架屏。
- 骨架屏应保持最终布局尺寸，避免数据返回后页面大幅跳动。
- 国家对选择器在候选对象尚未加载完成时置灰。

#### 3.8.2 慢加载

- 如果数据请求超过 8 秒仍未完成，页面展示提示：“数据加载较慢，请稍后或重试。”
- 慢加载提示应提供“重试”操作。
- 慢加载期间保留页面框架，不跳转到空白页。

#### 3.8.3 API 错误

- API 报错时保留顶部标题、国家对选择器和页面结构。
- 图表区域展示错误提示：“数据暂时无法加载。”
- 提供“重试”按钮。
- 如果存在上一次成功加载的数据，可继续展示旧数据，并标注“数据可能不是最新”。

#### 3.8.4 无数据组合

- 如果用户选择的关系组合完全没有可用事件数据，页面展示：“当前组合暂无足够事件数据。”
- 同时建议用户切换到首页重点关系。
- 不展示空白图表，不生成趋势段解释。

#### 3.8.5 数据不足

- 如果某个关系组合的数据覆盖少于 30 天，页面展示：“数据积累不足，趋势仅供参考。”
- 数据不足时仍可展示趋势，但不得强调 Top 趋势段结论。

#### 3.8.6 无明显趋势段

- 如果近 90 天未检测到明显趋势段，不强行标注趋势段。
- 解释区展示：“近 90 天未检测到明显趋势段。”
- 仍保留趋势图和重点关系信息。

#### 3.8.7 候选池配置异常

- 如果候选池少于 2 个可选对象，禁用国家对选择器。
- 页面展示：“候选对象配置不足，暂时无法选择关系组合。”
- 该状态视为配置异常，应在部署检查中被发现。

### 3.9 响应式策略

首版采用移动优先设计，同时保证桌面端信息密度。

移动端要求：

- 页面采用纵向布局。
- 重点关系卡片横向滑动展示。
- 趋势图全宽展示，优先保证日期、50 中性线和趋势段可读。
- 趋势段点击目标必须足够大，不能只依赖 hover。
- 趋势段解释区位于图表下方。
- 重点报道卡片纵向排列。

桌面端要求：

- 重点关系卡片使用网格布局。
- 趋势图与趋势段解释区可左右分栏展示。
- 方法说明可放在页面底部或侧边折叠区域。
- 桌面端可以展示更完整的日期刻度和辅助说明，但不增加额外核心指标。

## 4. 数据口径

### 4.1 数据源

首版读取共享 PostgreSQL 数据库：

```text
postgresql://<user>:<password>@localhost:5432/mapnews
```

文档、代码示例和界面中不得暴露真实数据库密码。

主要读取 MapNews 已导入和清洗后的 GDELT 事件数据。首版只依赖本地数据库，不额外抓取外部新闻网页。

### 4.2 时间范围

首版基于当前本地库可用的约 90 天数据展示，不要求历史回填。

页面应显示数据覆盖范围和最后更新时间，例如：

```text
数据范围：2026-02-23 至 2026-05-30
```

如果数据库覆盖不足 30 天，页面应展示“数据积累不足，趋势仅供参考”的提示。

### 4.3 关系温度

关系温度为 0-100 分：

- 50：中性。
- 50 以上：偏合作。
- 50 以下：偏冲突。

底层使用 GDELT Goldstein 冲突-合作事件信号映射到 0-100，但默认不在主界面展示 Goldstein 术语。

GoldsteinScale 本身是 `-10` 到 `+10` 的冲突-合作连续量表，GDELT Codebook 将其用于表示 CAMEO 事件类型的合作或冲突强度。关系温度不是新的学术量表，而是将该信号产品化为大众更容易理解的 0-100 展示。

参考依据：

- GDELT Codebook：GoldsteinScale 为 CAMEO 事件码分配 `-10` 到 `+10` 的冲突-合作强度分值。
- Goldstein, 1992：该量表用于将离散国际事件聚合为连续的冲突-合作时间序列。
- 相关 GDELT 双边关系研究常用时间窗口内的平均 Goldstein 分值衡量双边关系波动；部分研究使用 `NumMentions` 对 Goldstein 做加权平均，以降低单条事件噪音。

首版产品化映射方式为：

```text
relationship_temperature = clamp(50 + rolling_14d_goldstein * 12, 0, 100)
```

其中：

- `rolling_14d_goldstein` 为 14 日平滑后的报道加权 Goldstein 分值。
- 映射系数使用 `12`，用于放大近期趋势可读性，避免多数关系长期挤在 50 附近。
- `rolling_14d_goldstein = 0` 对应 50。
- `rolling_14d_goldstein <= -4.17` 会被截断为 0。
- `rolling_14d_goldstein >= +4.17` 会被截断为 100。

页面文案应将该指标称为“关系温度”。方法说明中应写明：该指标由 GDELT Goldstein 冲突-合作事件信号映射而来，是媒体事件信号，不是官方外交关系评分。

### 4.4 报道加权

单条事件对关系温度的影响按报道量加权，但不直接使用原始报道量作为线性权重。原因是新闻转载和重复报道可能让少数热点事件过度支配趋势线。

建议事件权重：

```text
event_weight = ln(1 + max(num_mentions, num_articles, 1))
```

其中 `ln` 为自然对数。工程实现中应固定使用自然对数，避免不同对数底数导致权重口径不一致。

每日关系分：

```text
daily_weighted_goldstein =
  sum(goldstein_scale * event_weight) / sum(event_weight)
```

这样既保留“被更多报道提及的事件更重要”的直觉，也通过对数权重降低重复转载造成的放大效应。

如果某日事件数或总权重过低，页面应降低该日解释强度，或在方法说明中提示“数据较少，趋势仅供参考”。

### 4.5 14 日平滑

趋势线先对每日报道加权 Goldstein 分值做 14 日滚动平均，再映射为 0-100 关系温度：

```text
rolling_14d_goldstein =
  avg(daily_weighted_goldstein over current day and previous 13 days)

relationship_temperature =
  clamp(50 + rolling_14d_goldstein * 12, 0, 100)
```

选择 14 日平滑的原因：

- 比单日分数更稳定。
- 比 28 日平滑更适合当前约 90 天数据。
- 对大众读者更容易解释为“近两周趋势”。

近 7 日变化状态使用当前 14 日平滑关系温度与 7 天前的 14 日平滑关系温度比较：

```text
delta_7d = relationship_temperature[t] - relationship_temperature[t - 7]
if delta_7d >= 5: 改善
if delta_7d <= -5: 恶化
otherwise: 平稳
```

该状态用于重点关系卡片和关系详情摘要，不替代 最多 6 个趋势段检测。

### 4.6 趋势段

系统自动识别每组关系中最重要的最多 6 个趋势段。

建议口径：

- 计算 14 日平滑后的关系温度。
- 比较当前点与 7 天前的 14 日平滑温度差。
- 检测从第 21 天之后开始，保证比较双方都已有完整 14 日平滑窗口，避免开头波动被误判为趋势段。
- 将连续同方向、时间上相互重叠的显著 7 日窗口合并为一个完整趋势段。
- 合并后的趋势段边界吸附到该段内的局部高点/低点，使高亮线段贴合用户看到的实际上升或下降段。
- 按合并后趋势段的总变化幅度排序，最多展示 6 段。

计算方式：

```text
segment_delta[t] = relationship_temperature[t] - relationship_temperature[t - 7]
if abs(segment_delta[t]) >= 5: 进入候选窗口
merge overlapping windows with the same direction
snap segment boundaries to local extrema
max_segments = 6
```

这样既能避免每天一个重叠窗口造成图表拥挤，也能让用户选中一整段肉眼可见的上升或下降趋势。

趋势段包括：

- 日期。
- 当日 14 日平滑温度。
- 与 7 天前相比的变化值。
- 方向：改善、恶化。
- 日期含义：该日期是变化被观察到的日期，不一定是单一事件发生日。

### 4.7 趋势段解释窗口

点击趋势段后，解释“变化期相对基准期发生了什么”。解释窗口独立于趋势平滑窗口，使用 7 天对比，以便让原因解释更聚焦。

推荐实现：

- 变化期：趋势段结束日期及其前 6 天，共 7 天。
- 基准期：变化期之前的 7 天。
- 不使用趋势段之后的未来数据，避免实时产品中的未来信息偏差。

```text
变化期 = t - 6 至 t
基准期 = t - 13 至 t - 7
```

解释区展示的是“截至趋势段结束日，最近 7 天相比之前 7 天的事件构成变化”。这样既保留 14 日趋势线的稳定性，也让原因解释聚焦于更近的事件和报道线索。

### 4.8 驱动事件类型

驱动事件类型基于 GDELT CAMEO 事件根类映射为用户可读分类。

主界面展示中文事件类型，例如：

- 公开声明。
- 磋商/外交接触。
- 外交合作。
- 物质合作。
- 让步/妥协。
- 反对/批评。
- 拒绝。
- 威胁。
- 军事姿态。
- 关系降级。
- 攻击。
- 战斗。

解释区只展示贡献最大的前三类，避免信息过载。

## 5. 数据与系统架构

### 5.1 项目边界

新产品在 `/Users/tianhaoran/Documents/国际关系趋势` 独立开发。

不得改动 `/Users/tianhaoran/Desktop/MapNews` 项目文件夹。

可以读取同一个 PostgreSQL 数据库。数据库层允许在不影响 MapNews 使用的前提下做轻量增强。

### 5.2 推荐技术栈

首版使用 Next.js 独立实现。

建议结构：

- Next.js App Router。
- 服务端 API 读取关系产品缓存表。
- 前端使用轻量图表组件展示趋势图。
- 定时预计算脚本每日更新缓存。

### 5.3 性能策略

由于 `gdelt_events_clean` 数据量较大，且按 actor 国家代码实时聚合会给 VPS 和数据库带来压力，首版不建议每次用户访问都直接扫描原始清洗表。

推荐方案：

1. 新增关系产品专用缓存表。
2. 每日定时任务从 GDELT clean 表读取数据并预计算。
3. 缓存候选池合法组合的趋势、趋势段和解释。
4. 前端/API 只读取缓存表。
5. 在必要时增加不会破坏 MapNews 的查询索引。

### 5.4 缓存内容

每日预计算应覆盖候选池内所有合法两两组合，包括重点关系和用户可选择的其他组合；`europe_gbr` 和 `europe_deu` 不进入预计算组合。

缓存至少包含：

- 关系组合 ID。
- 对象 A、对象 B。
- URL 分享代码，例如 `chn_usa`、`chn_europe`。
- 数据日期。
- 每日关系温度。
- 14 日平滑关系温度。
- 关系温度分区和状态文案。
- 当日事件数。
- 当日报道权重。
- 最多 6 个趋势段。
- 每个趋势段的解释摘要。
- 每个趋势段的前三类驱动事件。
- 每个趋势段的重点报道线索。
- 数据生成时间。

### 5.5 数据库增强边界

允许的数据库改动：

- 新增关系产品自己的缓存表。
- 新增必要查询索引。
- 新增只读预计算任务所需的元数据表。

不允许的数据库改动：

- 修改 MapNews 已有表字段含义。
- 删除或重建 MapNews 表。
- 影响 MapNews 导入流程的写入逻辑。
- 把关系产品逻辑耦合进 MapNews 项目代码。

### 5.6 定时更新

首版采用每日定时更新。

推荐更新流程：

1. MapNews 完成日度 GDELT 导入。
2. 关系产品定时任务读取最新 clean 数据。
3. 重新计算候选池合法组合的 90 天趋势。
4. 写入关系产品缓存表。
5. 前端显示新的数据更新时间。

如果当天数据缺失，页面保留上一次成功缓存，并显示“数据更新延迟”的提示。

## 6. 重点报道与解释质量

### 6.1 噪音过滤

GDELT 自动编码可能出现噪音，例如地名、组织名、人物名被错误映射到国家关系。

首版解释层应做基础过滤：

- 重点报道 URL 或来源文本中应出现相关国家、地区或核心政治实体关键词。
- 明显与所选关系无关的本地新闻应降低排序或过滤。
- 如果重点报道可信度不足，应减少展示数量，而不是强行凑满。

### 6.2 排序原则

重点报道排序综合考虑：

- 与关系双方的文本相关性。
- 事件影响方向是否与趋势段方向一致。
- 事件 Goldstein 绝对强度。
- 报道提及量或文章量。
- 来源 URL 是否可读。

### 6.3 模板解释

首版使用模板生成一句话解释，不接入 AI。

模板示例：

```text
截至该趋势段，{关系名}关系温度{上升/下降}。最近 7 天相比前 7 天，“{事件类型1}”“{事件类型2}”“{事件类型3}”等{偏合作/偏冲突}事件增加。
```

如果驱动事件不明显，则展示：

```text
这段时间关系温度出现变化，但现有结构化事件无法给出单一明确原因，以下是变化期间较相关的事件和报道线索。
```

## 7. 验收标准

首版需求实现后，应满足以下验收标准：

- 用户可以从重点关系卡片中选择中美、中俄、中欧、中日、中印、美伊、美俄。
- 用户可以通过两个下拉框选择候选池内任意两方合法组合。
- 用户可以通过 `/trend?pair=chn_usa` 直接打开指定关系；切换关系后 URL 同步更新，分享按钮复制当前 URL。
- URL pair 规则能区分非法组合回退默认关系、合法但无缓存组合展示空态。
- 页面能展示所选关系近 90 天的 0-100 关系温度趋势。
- 重点关系卡片文字状态采用 3 档折叠，颜色按 5 档温度分区渲染。
- 近 7 日变化状态按 `>= +5` 改善、`<= -5` 恶化、其余平稳判定。
- 趋势线使用 14 日平滑，并标出 最多 6 个趋势段。
- 点击趋势段后，页面展示变化期 7 天相对基准期 7 天的对比解释。
- 趋势段解释包含一句话摘要、前三类驱动事件和 3-6 条重点报道线索。
- 页面清楚展示数据范围和更新时间。
- 页面提供折叠式数据来源与限制说明。
- 主界面不默认暴露 GDELT、Goldstein、CAMEO 等术语。
- 文案不把媒体事件信号写成官方外交结论或确定因果。
- 页面进入后立即显示骨架屏；数据请求超过 8 秒时展示慢加载提示和重试操作。
- API 报错时保留页面框架，展示错误提示和重试按钮。
- 无数据组合、数据不足、无明显趋势段和候选池配置异常均有明确提示。
- 移动端采用纵向布局，重点卡片可横向滑动，趋势图和趋势段在手机上可读可点。
- 文档中能读到 PizzINT 是地缘关系/风险监控仪表盘类产品，以及本产品只参考其界面组织方式。
- 新产品不改动 MapNews 项目文件夹。
- 数据库改动仅限关系产品专用缓存表和必要索引，不破坏 MapNews 使用。

## 8. 后续路线

### 8.1 历史数据回填

当首版交互验证通过后，可回填一年或多年历史数据，支持更长期趋势和更稳定基线。

### 8.2 新闻标题与摘要

后续可在用户点击趋势段时按需抓取新闻标题，或定时补充标题缓存。

需要注意：

- 抓取失败时不能影响主功能。
- 标题和摘要必须标明来源。
- 不应长篇复制新闻正文。

### 8.3 AI 归纳

未来可基于重点报道标题、事件类型和来源链接生成更自然的解释。

AI 归纳应保持谨慎措辞，并展示“基于结构化新闻事件信号”的提示。

### 8.4 异常预警

在关系温度之外，可增加“紧张度”或“异常波动”指标，形成更接近 PizzINT 的监控能力。

但该能力不作为首版核心。

### 8.5 更多国家与组合

后续可扩展候选池，支持更多国家、地区、组织，甚至全球任意国家对。

扩展前需要处理：

- 国家名称和代码映射。
- 数据稀疏问题。
- 聚合对象口径。
- 查询和缓存规模。

## 9. 关键假设

- 首版界面语言为中文，来源链接保留原文。
- 当前产品只使用本地 MapNews 共享数据库中的结构化 GDELT 事件。
- 当前数据库约覆盖近 90 天数据，首版不要求长期历史基线。
- “任意两国”在首版中指候选池内任意两方合法组合。
- “欧盟/欧洲”是聚合对象，不是单一国家。
- PizzINT 是界面简洁性和监控节奏参考，不复制其威胁指数定位。
