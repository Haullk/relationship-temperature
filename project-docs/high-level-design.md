# 关系温度计一阶段概要设计

## 1. 设计目标

关系温度计的一阶段目标是基于现有 MapNews PostgreSQL 数据库，构建一个独立的双边关系趋势 Web 产品。用户可以在候选池内选择任意两方合法组合，查看近 90 天关系温度趋势，并点击 Top 趋势段查看简明原因解释和重点报道线索。

概要设计遵循以下原则：

- 新产品独立开发，不改动 `/Users/tianhaoran/Desktop/MapNews` 项目文件夹。
- 复用 MapNews 已有 GDELT clean 数据，不重复导入原始 GDELT。
- 通过关系产品专用缓存表降低线上查询压力。
- 模块职责清晰，尽量可以独立开发、独立测试。
- 首版面向大众读者，优先保证“看懂关系变化”和“知道为什么变”。

## 2. 总体架构

一阶段划分为五个模块：

```text
数据库基础
  ↓
数据装载
  ↓
数据处理
  ↓
服务 API
  ↓
前台页面
```

模块说明：

| 模块 | 核心职责 | 是否依赖 MapNews 代码 |
| --- | --- | --- |
| 数据库基础 | 提供原始 clean 数据和关系产品缓存表 | 不依赖 MapNews 代码，只读 MapNews 数据库表 |
| 数据装载 | 从数据库读取候选关系所需事件数据 | 不依赖 |
| 数据处理 | 计算关系温度、趋势段、解释和重点报道线索 | 不依赖 |
| 服务 API | 向前台提供候选对象、关系详情、状态信息 | 不依赖 |
| 前台页面 | 展示卡片、趋势图、解释区、状态和分享 URL | 不依赖 |

### 2.1 运行时选择

首版采用双运行时架构：

- 数据装载、数据处理、每日预计算任务运行在 Python，复用现有原型脚本思路，通过独立 venv 和每日 cron 运行。
- 服务 API 和前台页面运行在 Next.js/TypeScript。
- Python 任务只读取 GDELT clean 数据并写入关系产品缓存，不服务线上请求。
- 线上用户请求只经过 Next.js 服务 API 读取缓存，避免访问时实时扫描原始事件表。
- 现有 `scripts/plot_bilateral_relationships.py` 和 `scripts/analyze_turning_points.py` 属于探索原型，仍包含旧参数口径；生产预计算代码可复用其查询和分析思路，但必须按本文档口径重写或重构，不得直接沿用旧参数。

## 3. 模块设计

### 3.1 数据库基础模块

#### 职责

数据库基础模块负责提供数据存储边界：

- 读取 MapNews 已有 `gdelt_events_clean` 数据。
- 新增关系产品专用缓存表或缓存结构。
- 可增加必要索引，但不得破坏 MapNews 既有表结构和导入流程。

#### 输入

- MapNews PostgreSQL 数据库连接。
- `gdelt_events_clean` 中的事件日期、actor 国家代码、Goldstein、报道量、来源 URL 等字段。

#### 输出

- 关系产品缓存数据，供服务 API 读取。
- 数据更新时间、数据覆盖范围等元信息。

#### 设计边界

- 不删除、不重建、不修改 MapNews 已有表字段含义。
- 不把关系产品逻辑写入 MapNews 项目代码。
- 数据库增强只允许新增关系产品独立表、元数据表和必要索引。

#### 可测试点

- 缓存表可独立创建和清理。
- MapNews 现有表在新增关系产品对象后仍可正常查询。
- 缓存数据能记录生成时间和覆盖日期范围。

### 3.2 数据装载模块

#### 职责

数据装载模块负责把数据库中的 GDELT clean 事件转换为数据处理模块可使用的标准输入。

它不负责计算关系温度，只负责：

- 读取候选对象池配置文件 `config/candidate-pool.json`。
- 展开聚合对象，例如“欧盟/欧洲”。
- 生成候选池合法组合，并排除聚合对象与其成员对象的重叠组合。
- 根据关系组合查询近 90 天相关事件。
- 过滤明显无效记录，例如缺少日期、双方代码或 Goldstein 的事件。

#### 输入

- 候选对象池：
  - `CHN`、`USA`、`RUS`、`JPN`、`IND`、`IRN`、`TWN`、`GBR`、`DEU`、`europe`。
- 聚合对象配置：
  - `europe = EUR + GBR + DEU + FRA + ITA + ESP + NLD`。
- 合法组合排除规则：
  - `europe` 不与首版候选池中的 `GBR`、`DEU` 组成合法关系，避免欧洲聚合成员被重复计数。
- 时间范围：
  - 默认近 90 天。
- 数据库连接。
- 候选池配置文件：
  - `config/candidate-pool.json`。

#### 输出

标准化事件列表，至少包含：

- 事件日期。
- 对象 A、对象 B。
- `goldstein_scale`。
- `num_mentions`。
- `num_articles`。
- 事件根类。
- actor 名称。
- 来源域名。
- 来源 URL。

#### 设计边界

- 只做查询和轻量标准化，不计算最终趋势。
- 不做解释层噪音过滤，文本相关性过滤属于数据处理模块。
- 不直接服务前端。
- 不进行网页抓取。

#### 可测试点

- 给定 `chn_usa`，能正确查询中美双向事件。
- 给定 `chn_europe`，能正确展开欧洲聚合对象。
- `europe_gbr` 和 `europe_deu` 不会进入合法组合列表。
- 同一对象不能组成关系组合。
- 无数据组合返回空事件列表，而不是抛出不可恢复错误。

### 3.3 数据处理模块

#### 职责

数据处理模块负责把标准化事件转换为产品可展示的数据。

核心计算包括：

- 每日关系温度。
- 14 日平滑关系温度。
- 温度分区和状态文案。
- 最多 6 个趋势段。
- 变化期 7 天相对基准期 7 天的解释。
- 重点报道线索。

#### 输入

- 数据装载模块输出的标准化事件列表。
- 关系组合配置。
- 温度分区配置。

#### 输出

关系组合的缓存结果：

- 关系组合 ID，例如 `chn_usa`。
- 关系显示名，例如“中美”。
- 数据覆盖范围。
- 每日趋势点。
- 当前关系温度和状态。
- 近 7 日变化状态。
- 最多 6 个趋势段。
- 趋势段状态原因，例如 `normal`、`data_insufficient`、`no_significant_turning_points`。
- 每个趋势段的解释摘要、驱动事件类型和重点报道线索。

#### 核心算法口径

事件权重：

```text
event_weight = ln(1 + max(num_mentions, num_articles, 1))
```

每日报道加权 Goldstein：

```text
daily_weighted_goldstein =
  sum(goldstein_scale * event_weight) / sum(event_weight)
```

14 日平滑：

```text
rolling_14d_goldstein =
  avg(daily_weighted_goldstein over current day and previous 13 days)
```

关系温度：

```text
relationship_temperature = clamp(50 + rolling_14d_goldstein * 12, 0, 100)
```

映射系数首版使用 `12`，这是产品化校准参数，用于放大近 90 天趋势可读性；`rolling_14d_goldstein = 0` 仍对应中性 50，约 `<= -4.17` 或 `>= +4.17` 时分别截断到 0 或 100。

趋势段：

```text
segment_delta[t] = relationship_temperature[t] - relationship_temperature[t - 7]
if abs(segment_delta[t]) >= 5: 进入候选窗口
merge overlapping windows with the same direction
snap segment boundaries to local extrema
max_segments = 6
```

趋势段候选从第 21 天之后开始检测，保证比较双方都已有完整 14 日平滑窗口。连续同方向、时间上重叠的显著 7 日窗口合并为一个完整趋势段，边界吸附到局部高点/低点。

近 7 日变化状态：

```text
delta_7d = relationship_temperature[t] - relationship_temperature[t - 7]
if delta_7d >= 5: 改善
if delta_7d <= -5: 恶化
otherwise: 平稳
```

解释窗口：

```text
变化期 = t - 6 至 t
基准期 = t - 13 至 t - 7
```

解释层文本相关性过滤：

- 驱动事件统计使用 actor 名称、来源域名、来源 URL 的组合文本做相关性过滤，要求出现关系双方对应的国家、地区或核心政治实体关键词。
- 重点报道线索使用更严格的来源文本过滤，要求来源域名或来源 URL 中出现相关关键词。
- 噪音过滤属于数据处理模块，不放在数据装载模块，避免把解释质量规则混入基础查询。

#### 温度分区

| 温度范围 | 状态文案 |
| --- | --- |
| 0-35 | 明显偏冲突 |
| 35-45 | 偏冲突 |
| 45-55 | 接近中性 |
| 55-65 | 偏合作 |
| 65-100 | 明显偏合作 |

#### 设计边界

- 不访问前端状态。
- 不直接写页面。
- 不抓取新闻标题或正文。
- 不使用 AI 生成解释。

#### 可测试点

- 同一批事件输入可得到稳定、可重复的关系温度。
- `ln` 权重使用自然对数。
- 14 日平滑和 7 日趋势段检测分别定义，不混用窗口含义。
- 近 7 日变化阈值固定为 `+5/-5`。
- 显著 7 日窗口会合并为最多 6 个完整趋势段，趋势段边界吸附到局部高点/低点。
- 数据不足 30 天时返回 `turning_point_status = data_insufficient`。
- 有足够数据但无明显趋势段时返回 `turning_point_status = no_significant_turning_points`。

### 3.4 服务 API 模块

#### 职责

服务 API 模块负责把缓存数据转换为前台页面需要的接口响应。

首版 API 不直接扫描 `gdelt_events_clean`，而是读取关系产品缓存表或缓存结果。

#### 输入

- 前端请求参数：
  - `pair`，例如 `chn_usa`。
- 缓存表中的关系趋势与解释数据。

#### 输出

前端页面数据：

- 候选对象列表。
- 重点关系卡片数据。
- 当前关系详情。
- 趋势点列表。
- 趋势段列表。
- 趋势段解释详情。
- 数据更新时间。
- 状态信息，例如无数据、数据不足、缓存过期；缓存过期的业务状态码可使用 `stale_cache`。
- 缓存状态，例如 `fresh`、`stale`、`missing`。

#### API 行为

- `pair` 缺失时返回默认关系 `chn_usa`。
- `pair` 校验顺序为：格式解析 → 规范化组合 ID → 候选池合法组合校验 → 缓存读取。
- 规范化组合 ID 使用两个小写对象代码按字母序排序后以 `_` 连接；例如 `usa_chn` 等同于 `chn_usa`，API 返回规范 ID，前台 URL 也回写规范 ID。
- `pair` 不在 `config/candidate-pool.json` 可生成的合法组合列表中时，视为非法，回退默认关系 `chn_usa`，并附带轻量提示。
- `europe_gbr` 和 `europe_deu` 按非法组合处理。
- `pair` 合法但缓存表没有该组合数据时，返回该组合的无数据空态，不回退默认关系。
- 缓存过期但仍有可用旧数据时，返回 `stale` 状态，让前台展示“数据可能不是最新”。
- 缓存缺失且无法提供旧数据时，返回 `missing` 状态，不让前端猜测原因。
- API 响应不暴露数据库连接信息。

#### 设计边界

- 不做复杂计算，只做缓存读取、参数校验和响应组装。
- 不负责页面交互。
- 不抓取外部网页。

#### 可测试点

- `/trend?pair=chn_usa` 能拿到中美关系数据。
- `/trend?pair=usa_chn` 能归一化为 `chn_usa`，不回退默认关系。
- 无效 `pair` 能回退默认关系。
- 合法 `pair` 但缓存无数据时返回该组合空态，而不是回退默认关系。
- 缓存过期时返回 `stale` 状态和旧数据。
- 无数据组合返回空态所需信息。
- API 错误可被前端识别为错误态。

### 3.5 前台页面模块

#### 职责

前台页面模块负责面向用户展示关系温度计的核心体验。

页面必须支持：

- 重点关系卡片。
- 候选对象下拉选择。
- URL 分享。
- 单对关系趋势图。
- 90 日、30 日、15 日图表范围切换。
- hover 日期/温度 tooltip 与垂直辅助线。
- 关系温度分区配色。
- 最多 6 个趋势段点击。
- 趋势段解释区。
- 重点报道线索。
- 折叠式方法说明。
- 加载态、空态、错误态。

#### 输入

- 服务 API 返回的页面数据。
- URL 参数 `pair`。
- 用户点击和选择行为。

#### 输出

- 页面展示。
- 更新后的 URL 参数。
- 分享按钮复制的当前 URL。

#### 交互边界

- 页面不直接访问数据库。
- 页面不实现关系温度计算。
- 页面不依赖 hover，移动端趋势段必须可点击。
- 页面不生成虚假新闻标题。

#### 卡片状态表达

- 重点关系卡片文字状态采用 3 档折叠：`明显偏合作`、`偏合作` 显示为“偏合作”；`接近中性` 显示为“接近中性”；`偏冲突`、`明显偏冲突` 显示为“偏冲突”。
- 卡片背景、状态标签和迷你趋势线末端仍按 5 档温度分区渲染。
- 主趋势图、图例和趋势段继续使用完整 5 档温度文案。

#### 响应式策略

- 移动优先。
- 移动端纵向布局：
  - 重点关系卡片横向滑动。
  - 趋势图全宽。
  - 解释区在图表下方。
- 桌面端增强信息密度：
  - 卡片网格。
  - 趋势图和解释区左右分栏。

#### 状态设计

- 加载态：立即展示骨架屏。
- 慢加载：超过 8 秒提示并提供重试。
- API 错误：保留页面框架，展示错误提示和重试按钮。
- 数据更新延迟：展示旧数据，并提示“数据可能不是最新”。
- 无数据组合：提示暂无足够事件数据。
- 数据不足：提示趋势仅供参考。
- 无明显趋势段：不强行标点。
- 候选池少于 2 个对象：禁用选择器。

#### 可测试点

- URL 参数能正确选中关系组合。
- 切换关系后 URL 同步更新。
- 分享按钮复制当前 URL。
- 温度分区颜色与文字状态一致。
- 移动端不依赖 hover。
- 移动端和桌面端使用同一套趋势图与解释区组件，通过 CSS media query 或 container query 切换纵向/左右分栏布局，不维护两套组件树。
- 各类状态均可触发并显示正确文案。

## 4. 模块关系与数据流

### 4.1 日常预计算流程

```text
每日 Python cron 预计算任务
  → 数据装载模块读取 config/candidate-pool.json 和近 90 天 GDELT clean 事件
  → 数据处理模块计算趋势、趋势段、解释和报道线索
  → 数据库基础模块写入关系产品缓存表
  → 服务 API 读取最新缓存
  → 前台页面展示
```

### 4.2 用户访问流程

```text
用户打开 /trend?pair=chn_usa
  → 前台页面读取 URL 参数
  → 服务 API 校验 pair
  → 服务 API 读取缓存数据
  → 前台页面渲染卡片、趋势图、解释区
  → 用户点击趋势段
  → 前台页面展示该趋势段解释
```

### 4.3 模块依赖方向

依赖方向必须单向：

```text
前台页面 → 服务 API → 缓存数据
数据处理 → 数据装载 → 数据库基础
服务 API 不依赖数据处理运行时
前台页面不依赖数据库
```

这样可以保证：

- 数据处理可以离线测试。
- API 可以用缓存假数据测试。
- 前台可以用 mock API 测试。
- 数据库变更不会直接影响页面组件。

## 5. 一阶段交付边界

一阶段交付概要设计覆盖首版 MVP，不包含：

- 账号、订阅、提醒、告警。
- CSV、图片或报告导出。
- 外部网页标题或正文抓取。
- AI 自动总结。
- 一年以上历史数据回填。
- 全球任意国家对。

一阶段可以为后续能力保留扩展点：

- 更长历史数据。
- 新闻标题缓存。
- AI 解释。
- 异常预警。
- 更多国家和地区。

## 6. 测试策略

### 6.1 模块级测试

- 数据装载模块：测试 `config/candidate-pool.json` 读取、国家对查询、欧洲聚合、欧洲重叠组合排除、空数据组合。

### 6.3 验收标准

概要设计对应的 MVP 实现应满足：

- 模块边界清晰，不出现前台直接查库或 API 实时扫原始表。
- Python cron 预计算任务负责写入缓存，Next.js API 只读取缓存。
- `config/candidate-pool.json` 是候选池和合法组合生成的配置来源。
- 合法组合排除 `europe_gbr` 和 `europe_deu`，避免欧洲聚合对象与成员对象重复计数。
- 关系温度和趋势段口径与需求文档一致。
- 缓存表或缓存结果能支撑候选池合法组合。
- API 能区分非法 pair、合法但无缓存、缓存过期三类情况。
- 前台状态设计完整。
- 前台趋势图和解释区在移动端、桌面端复用同一组件树，通过 CSS 切换布局。
